# ADR 0005 — Self-signed TLS at the outer NLB for the POC

**Status:** Accepted
**Date:** 2026-04-27
**Deciders:** Principal Engineer

## Context

The eventual JPMC production deployment uses internal CA certificates
issued through the firm's PKI. We do not have access to that PKI in the
personal account where the POC runs. Two options:

1. Skip TLS at the outer NLB.
2. Issue self-signed certs and import to ACM.

Option 1 makes the POC topology diverge from JPMC at the network layer,
which we explicitly decided to avoid (SPEC §1.1: "100% topology mirror").

## Decision

Use **self-signed CA + leaf certs** generated by the Terraform `tls`
provider, imported to ACM in both regions. The outer NLB has a TLS
listener using these certs. The CloudWatch Synthetics canary runs with
`ignoreHttpsErrors: true` (Puppeteer flag) controlled by profile field
`canary.ignore_tls_errors`.

JPMC port: flip `ignore_tls_errors: false` and supply the internal CA
chain via `canary.internal_ca_bundle_s3_uri`.

## Consequences

**Positive:**
- Topology is identical to JPMC at the TLS termination boundary.
- Profile-driven; no code change to flip.
- The TLS-at-the-outer-NLB code path is exercised in every scenario.

**Negative:**
- Production code paths must NEVER skip TLS verification — the carve-out
  is canary-only.
- Operators interacting with the outer NLB from the personal account
  have to pass `-k` to curl etc. This is documented in
  `runbooks/RUNBOOK-manual-failover.md`.

## Alternatives Considered

- **Skip TLS:** Rejected — diverges from JPMC topology.
- **AWS Certificate Manager public certs:** Rejected — requires a public
  domain we own; the POC uses `failover.internal` which is private.
